# Структура и функционал проекта

Этот файл описывает дорефакторную версию приложения, которая лежит в GitHub до разбиения фронтенда на `static/css` и `static/js`.

Базовая версия для сравнения: коммит `9f91892`  
Смысл этого документа: быстро понять, из каких файлов состоял сервис, как он работал и где находилась основная логика.

## 1. Что это за сервис

`WB Analytics Dashboard` это аналитический сервис для продавца Wildberries.

Он решает три основные задачи:
- принимает данные по товарам из Excel-файла;
- либо загружает данные напрямую из `WB Seller Analytics API`;
- показывает по ним дашборд с KPI, матрицей решений, рекомендациями и ИИ-анализом.

Фронтенд в этой версии был собран в одном файле `dashboard.html`.  
Бэкенд работал на `FastAPI` и состоял из двух Python-файлов:
- `backend/main.py`
- `backend/wb_api_client.py`

## 2. Структура старой версии

```text
.
├── .gitignore
├── Procfile
├── README.md
├── dashboard.html
├── railway.json
├── requirements.txt
└── backend/
    ├── main.py
    └── wb_api_client.py
```

Важно: в этой версии `Node.js` не использовался вообще.  
Приложение запускалось только через `Python + FastAPI`.

## 3. Общая схема работы

Пользовательский поток выглядел так:

1. Пользователь открывает `dashboard.html`, который отдаёт `FastAPI`.
2. Пользователь либо загружает Excel, либо вводит ключ и период для `WB API`.
3. Бэкенд приводит данные к единой внутренней структуре.
4. Обработанные данные сохраняются во временное in-memory хранилище `DATA_STORE`.
5. Фронтенд запрашивает отдельные блоки:
   - summary
   - hits
   - outsiders
   - matrix
   - actions
6. Дашборд отрисовывает три вкладки:
   - Командный центр
   - Матрица решений
   - Action Dashboard
7. При необходимости пользователь запускает ИИ-анализ через Claude API.

## 4. Файлы и их роль

### `.gitignore`

Стандартный служебный файл Git.

Что делает:
- скрывает из git лишние локальные артефакты;
- не влияет на логику приложения.

### `README.md`

Короткое описание проекта.

Что в нём было:
- назначение дашборда;
- список основных возможностей;
- базовые шаги локального запуска;
- инструкция по деплою на Railway;
- описание переменной окружения `ANTHROPIC_API_KEY`.

Зачем нужен:
- даёт быстрый вход в проект;
- помогает понять, как запустить сервис без чтения кода.

### `requirements.txt`

Список Python-зависимостей.

Ключевые пакеты:
- `fastapi` и `uvicorn` для API и запуска сервера;
- `pandas` и `openpyxl` для чтения Excel;
- `python-multipart` для загрузки файлов;
- `httpx` для запросов к WB API;
- `anthropic` для ИИ-анализа.

Зачем нужен:
- полностью определяет Python-окружение приложения.

### `Procfile`

Команда запуска для платформ вроде Railway/Heroku-стиля.

Что делает:
- запускает `uvicorn` из директории `backend`;
- поднимает API на порту из переменной окружения `PORT`.

### `railway.json`

Конфигурация деплоя для Railway.

Что делает:
- задаёт способ сборки;
- задаёт команду запуска;
- указывает политику рестартов.

Зачем нужен:
- позволяет разворачивать сервис без ручной настройки команды старта.

### `dashboard.html`

Это главный файл старого фронтенда и самая тяжёлая часть приложения.

Что было внутри:
- вся HTML-структура страницы;
- все CSS-стили;
- весь JavaScript приложения.

По сути это был одновременно:
- шаблон интерфейса;
- файл стилей;
- модуль состояния;
- клиент API;
- слой рендера.

#### Что делал `dashboard.html`

1. Показывал шапку приложения.
2. Давал две точки входа для данных:
   - загрузка Excel;
   - загрузка через `WB API`.
3. Управлял вкладками:
   - `Командный центр`
   - `Матрица решений`
   - `Action Dashboard`
4. Хранил клиентское состояние в переменной `dashboardData`.
5. Вызывал backend API.
6. Рендерил таблицы, KPI, воронку, рекомендации и AI-блок.
7. Экспортировал matrix/actions в CSV.

#### Основные блоки внутри `dashboard.html`

`<style>`
- содержал всю визуальную систему;
- описывал layout, KPI-карточки, таблицы, вкладки, BCG-блоки, action cards, empty states, адаптивность.

HTML-разметка
- header;
- upload zone;
- панель `WB API`;
- табы;
- три контейнера под контент вкладок.

`<script>`
- определял `API_BASE`;
- держал глобальное состояние `dashboardData`;
- содержал форматтеры чисел, валют, процентов и динамики;
- обрабатывал загрузку файла и drag-and-drop;
- работал с `WB API`-панелью;
- загружал dashboard-данные с backend;
- рендерил все экраны;
- вызывал ИИ-анализ;
- делал экспорт в CSV.

#### Самые важные функции во фронтенде

`loadDashboard()`
- центральная клиентская функция;
- после загрузки данных дёргала сразу несколько backend-эндпоинтов;
- собирала полный объект данных для интерфейса;
- запускала рендер вкладок.

`renderCommandCenter()`
- строила главный экран с KPI, funnel, hits, outsiders и AI-блоком.

`renderMatrix()`
- строила BCG-матрицу;
- показывала 4 квадранта;
- позволяла экспортировать товары по каждой категории.

`renderActions()`
- показывала приоритетные действия;
- группировала их как `critical`, `important`, `opportunities`;
- показывала простой прогноз на следующую неделю.

`getAIAnalysis()`
- отправляла POST на `/api/ai/analyze`;
- получала текстовое заключение от Claude;
- вставляла его в UI.

`exportMatrixToExcel()` и `exportActionsToExcel()`
- фактически экспортировали не Excel, а CSV;
- формировали таблицу на клиенте и скачивали её через `Blob`.

#### Почему этот файл считался проблемным

Плюс:
- всё собрано в одном месте;
- быстро запустить и показать MVP.

Минусы:
- слишком много ответственности в одном файле;
- сложно искать нужную логику;
- опасно менять UI и рендер без побочных эффектов;
- трудно масштабировать;
- стили, API-вызовы и рендер связаны слишком плотно.

### `backend/main.py`

Это главный backend-файл и центр всей серверной логики.

Что он делал:
- создавал `FastAPI` приложение;
- включал `CORS`;
- принимал Excel-файлы;
- обрабатывал данные;
- считал агрегаты для dashboard;
- отдавал API для фронта;
- вызывал `wb_api_client.py`;
- вызывал Claude API для ИИ-анализа;
- отдавал `dashboard.html` по `/`.

#### Главные концепции внутри `main.py`

`DATA_STORE`
- глобальное in-memory хранилище;
- содержало `raw_data`, `processed_data`, дату загрузки, период и имя файла;
- не использовало базу данных.

Что это значило на практике:
- данные жили только в памяти процесса;
- после перезапуска сервера всё сбрасывалось;
- приложение было ориентировано на одного активного пользователя или демо-режим.

`COLUMN_MAPPING`
- таблица сопоставления русских колонок Excel во внутренние имена полей;
- это был фундамент для унификации Excel-данных.

Вспомогательные функции:
- `safe_float()` и `safe_int()` безопасно приводили значения;
- `calculate_dynamics()` считала изменение в процентах;
- `classify_product()` относила товар к `star/question/cash_cow/dog`;
- `process_data()` нормализовала Excel-данные и добавляла расчётные поля.

#### Основные API-эндпоинты

`POST /api/upload`
- принимал Excel-файл;
- читал его через `pandas.read_excel`;
- прогонял через `process_data()`;
- сохранял результат в `DATA_STORE`.

`GET /api/dashboard/summary`
- считал главные KPI;
- строил сводку по выручке, заказам, выкупам, отменам, остаткам;
- считал funnel и конверсии;
- возвращал главный блок данных для командного центра.

Это один из самых сложных кусков кода в старой версии, потому что именно здесь была зашита аналитическая логика:
- фильтрация товаров с заказами;
- суммирование текущего и прошлого периода;
- расчёт средневзвешенных показателей;
- формирование структуры `kpi` и `funnel`.

`GET /api/dashboard/hits`
- возвращал топ товаров по выручке.

`GET /api/dashboard/outsiders`
- искал проблемные товары;
- правила были простые и захардкоженные:
  - нет заказов при показах;
  - низкий CTR;
  - низкий процент выкупа.

`GET /api/dashboard/matrix`
- формировал BCG-матрицу;
- группировал все товары по 4 категориям;
- возвращал агрегаты и список товаров в каждой группе.

`GET /api/dashboard/actions`
- собирал управленческие рекомендации;
- делил их на:
  - `critical`
  - `important`
  - `opportunities`

Логика была rule-based:
- нулевой остаток при прошлых продажах;
- сильное падение выручки;
- низкий рейтинг карточки;
- высокий CTR при низкой конверсии;
- “звёзды” с малыми остатками;
- “вопросы” для тестирования.

`POST /api/fetch-from-wb`
- вызывал `fetch_sales_funnel()` из `wb_api_client.py`;
- затем прогонял ответ через `map_api_to_internal()`;
- после этого дообогащал данные расчётными полями так же, как Excel-поток.

Смысл этого маршрута:
- WB API и Excel должны были прийти к одному внутреннему формату;
- фронтенд дальше уже не думал, откуда взялись данные.

`POST /api/ai/analyze`
- собирал краткую summary-структуру;
- отправлял её в Claude через `Anthropic`;
- получал готовый текст анализа на русском.

`GET /api/products`
- отдавал полный список товаров с фильтрацией и пагинацией;
- в старом интерфейсе явно почти не использовался.

`GET /api/categories`
- отдавал категории и агрегаты по ним;
- тоже был скорее запасным сервисным маршрутом.

`GET /health`
- простой health-check.

`GET /`
- отдавал `dashboard.html`.

### `backend/wb_api_client.py`

Это изолированный клиент для Wildberries Seller Analytics API.

Что он делал:
- ходил в `Sales Funnel API v3`;
- поддерживал пагинацию;
- уважал rate limit;
- ограничивал допустимую глубину периода 365 днями;
- переводил ответ WB API во внутренний формат проекта.

#### Ключевые функции

`fetch_sales_funnel()`
- отправляла POST-запросы в WB API;
- ходила страницами по `limit/offset`;
- при `429` ждала и повторяла запрос;
- автоматически рассчитывала `past period`, если он не передан;
- ограничивала старые даты.

`map_api_to_internal()`
- самая важная функция этого файла;
- приводила внешний формат WB API к той же внутренней схеме, что и Excel.

Это был критичный архитектурный мост, потому что фронтенд и основной backend дальше ожидали одинаковые поля:
- `orders_value`
- `orders_qty`
- `purchase_rate`
- `card_views`
- `stock_wb`
- `stock_mp`
- и так далее

Без этой нормализации пришлось бы поддерживать две отдельные ветки логики:
- одну для Excel;
- вторую для WB API.

## 5. Основные пользовательские сценарии в старой версии

### Сценарий 1. Загрузка Excel

1. Пользователь выбирает файл в интерфейсе.
2. Фронтенд отправляет файл на `/api/upload`.
3. Бэкенд читает Excel и нормализует данные.
4. Фронтенд вызывает набор dashboard-эндпоинтов.
5. На странице появляются KPI, matrix, actions и AI-блок.

### Сценарий 2. Загрузка из WB API

1. Пользователь вводит ключ и диапазон дат.
2. Фронтенд отправляет запрос на `/api/fetch-from-wb`.
3. Бэкенд получает сырой ответ WB API.
4. `wb_api_client.py` переводит его во внутреннюю структуру.
5. Далее интерфейс работает так же, как после Excel.

### Сценарий 3. ИИ-анализ

1. После загрузки данных пользователь нажимает кнопку анализа.
2. Фронтенд отправляет запрос на `/api/ai/analyze`.
3. Бэкенд собирает сжатую summary-модель данных.
4. Claude возвращает текст рекомендаций.
5. Текст показывается в блоке AI.

## 6. Тяжёлые и важные моменты старой архитектуры

### Единая внутренняя схема данных

Это один из главных смысловых центров проекта.

И Excel, и WB API в итоге должны были превращаться в одинаковые записи.  
За это отвечали:
- `COLUMN_MAPPING`
- `process_data()`
- `map_api_to_internal()`

Благодаря этому dashboard-эндпоинты могли работать почти одинаково для обоих источников.

### Вся фронтенд-логика в одном файле

Это ускоряло старт, но создавало технический долг:
- трудно разделять ответственность;
- трудно менять дизайн отдельно от логики;
- трудно переиспользовать части кода;
- трудно тестировать изменения кусками.

### In-memory хранение без БД

`DATA_STORE` делал приложение простым, но имел ограничения:
- нет постоянного хранения;
- нет истории сессий;
- нет многопользовательской изоляции;
- после рестарта данные пропадают.

### Неидеальное совпадение Excel и WB API семантики

Хотя оба источника приводились к одной схеме, метрики не были полностью идентичны.

Например:
- в WB API нет отдельного полноценного поля показов, поэтому часть данных аппроксимировалась;
- названия некоторых конверсий и логика метрик в Excel и API семантически отличаются.

Это не ломало интерфейс, но важно понимать при развитии аналитики.

## 7. Что было хорошего в старой версии

- быстро запускается;
- понятная MVP-логика;
- единый интерфейс для Excel и WB API;
- уже есть базовый AI-слой;
- уже есть готовые аналитические блоки и простые управленческие рекомендации.

## 8. Главные ограничения старой версии

- монолитный `dashboard.html`;
- тесная связка UI и рендера;
- сложнее безопасно дорабатывать;
- нет БД и долговременного хранения;
- часть аналитических правил захардкожена;
- логика рекомендаций и классификации пока простая и rule-based.
